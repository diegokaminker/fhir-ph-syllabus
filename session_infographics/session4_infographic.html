FHIR References

Why References?

One resource is usually not enough. We need multiple resources to represent relationships and complex healthcare scenarios. References connect resources together to create meaningful clinical information.

Reference Types

  • Absolute: Full URL to specific server
  • Relative: Reference within same server
  • Contained: Resource within same resource
  • Identifier: Human identifier reference
  • Bundled: Reference within same Bundle

Reference Examples

  • Absolute: http://myserver/fhir/Patient/20911
  • Relative: Patient/20911
  • Contained: #MyPatient
  • Identifier: System + Value
  • Bundled: UUID references

Contained Resources

  • When there is no external identity
  • Has limitations - no reuse
  • Very hard to resolve to 'real'
  • Use when must include information
  • No independent existence

FHIR Terminology

Key to Semantic Interoperability

Terminology deals with coded elements and the meaning of concepts. Interoperability = Syntax + Structure + Semantics. FHIR provides comprehensive terminology support through Code Systems, Value Sets, and Bindings.

Coded Data Types

  • code: Simple coded value (e.g., "male")
  • CodeableConcept: Complex coded concept
  • Quantity: Numeric value with units
  • Coding: System + Code + Display

Terminology Components

  • Code System: Defines set of concepts
  • Value Set: Selection of codes for context
  • Binding: Links element to Value Set
  • Display: Human-readable text

Terminology Services

  • $expand: Expand Value Set
  • $lookup: Get display for code
  • $validate-code: Validate code
  • $translate: Translate between systems

HL7 V2 to FHIR Mapping

Why V2 to FHIR Conversion?

HL7 V2.x is still the lingua franca of healthcare despite its dialects. To get ALL clinical information from existing systems into FHIR-based repositories, the easiest way is to 'listen' to existing V2 feeds: ADT, Lab Results, Immunizations.

Mapping Process

  • Official mapping for common messages
  • ADT, Orders, Results, Scheduling
  • Not a tool - it's a mapping specification
  • Custom scripts often needed
  • Each V2 implementation is unique

Mapping Challenges

  • V2 implementations are opinionated
  • Extensions and customizations common
  • Different versions and dialects
  • No one-size-fits-all solution
  • Requires understanding of both standards

Mapping Benefits

  • Access to legacy system data
  • Gradual migration to FHIR
  • Preserve existing workflows
  • Enable modern FHIR applications
  • Bridge between old and new systems
